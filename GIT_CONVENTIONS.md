# Git 컨벤션

브랜치 · 커밋 · 태그 규칙을 통일해 리뷰와 릴리스를 단순하게 만듭니다.  
현재 커밋 로그에서 쓰는 `type(scope): 설명` 형식을 기준으로 정리했습니다.

## 브랜치 전략

| 브랜치 | 목적 | 설명 |
|--------|------|------|
| `main` | 운영(또는 배포 기준) 최신 | 항상 배포 가능하거나 방금 배포된 코드를 가리키도록 유지 |
| `develop` | 개발 통합 | 기능이 모이는 기본 개발 브랜치(팀에서 도입 시) |
| `release/*` | 릴리스·스테이징 | QA·스테이징 배포용. 예: `release/v1.2.0` |
| `feature/*` | 기능 개발 | 예: `feature/login-api`, `feature/store-search` |

### 네이밍 예시

- 기능: `feature/community-post-api`, `feature/map-store-filter`
- 릴리스: `release/v1.2.0`, `release/hotfix-payment`

자동화나 보조 작업이 있더라도, 기본 작업 브랜치는 `feature/*`, `fix/*`, `chore/*`로 통일합니다.

## 커밋 메시지

### 언어

- **커밋 로그(제목·본문)는 반드시 한글로 작성합니다.** PR·MR 설명도 동일 원칙을 따르는 것을 권장합니다.
- `type`, `scope` 같은 접두 형식은 아래 형식을 유지하고, 콜론 뒤 설명은 한글로 씁니다.

### 형식

```
type(scope): 짧은 설명 (한 줄, 명령형 또는 현재형)
```

- **scope**: 변경이 큰 영역. 모노레포 기준으로 `client`, `server`, `docs` 등.
- 본문이 필요하면 한 줄 띄우고 `Why`, `What` 순으로 보강.

### 실제 로그 예시

- `feat(client): 토스 쇼핑·커뮤니티 웹 프론트 빌드 추가`
- `chore(server): gradle.properties로 버전 1.0.0 관리`

### type

| 타입 | 의미 |
|------|------|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `docs` | 문서만 |
| `style` | 포맷·세미콜론·공백 등 (동작 변화 없음) |
| `refactor` | 리팩터링 (동작 유지) |
| `test` | 테스트 추가·수정 |
| `chore` | 빌드·설정·잡무 |

## 권장 워크플로우

1. `develop`(또는 합의된 기본 브랜치)에서 `feature/...` 생성  
2. 컨벤션에 맞게 커밋  
3. PR(Merge Request)로 리뷰  
4. `develop`에 머지 → 필요 시 `release/...`에서 스테이징  
5. 검증 후 `main`에 반영(머지 또는 패치)

## 버전 태그 (`v*`)

- **릴리스 지점**에만 `v1.0.0`처럼 태그를 둡니다. 매 커밋마다 태그할 필요는 없습니다.
- 한 번 올린 태그는 **개발이 이어진다고 해서 삭제할 필요 없습니다.** 이미 clone·CI가 참조했을 수 있습니다.
- 잘못 찍은 태그만 정리할 때는 원격·로컬 모두 조정과 팀 공지가 필요합니다.

## 참고

- GitHub Pull Request / GitLab Merge Request 템플릿 활용 권장  
- 커밋은 “무엇을·왜” 한 줄에 드러나게 작성(위 언어 규칙 준수)

# Design: 상점 데이터 리빌드 및 `works` 카탈로그 기준 LLM 매핑

## 목적

상점 데이터를 **CSV 기반 파이프라인 전체 재실행**으로 새로 쌓되, 작품 연결(`shop_works`)은 **이미 존재하는 MySQL `works` 행**만 대상으로 하고, 블로그 맥락과 카탈로그를 함께 본 **LLM 추론**으로 `work_id`를 고른다. 매칭되지 않는 작품명은 **연결을 생략**한다.

## 전제·합의 사항

| 항목 | 결정 |
|------|------|
| 상점 입력 | 기존과 동일 **A 경로**: `shop.csv`(등) → 블로그 URL 보강 → 크롤 → Gemini 정제 |
| `works` DDL | **이미 생성되어 있음**. 본 설계는 스키마 신설이 아니라 파이프라인·매칭 동작을 정의한다. |
| `works` 데이터 | **기존 데이터를 유지**한다. 추가·수정은 **운영자 수동**으로 한다. AniList 동기 등은 본 리빌드의 필수 전제가 아니다. |
| 매칭 실패 | 해당 작품은 **`shop_works`에 넣지 않고 생략**. 상점의 나머지 필드·카테고리 등 저장은 정책에 따라 진행. |
| 별칭 소스 | **`works` 테이블에 들어 있는 제목 계열 컬럼만** 매칭 후보 문자열로 사용한다. 별도 `work_aliases` 테이블·LLM 생성 별칭 캐시는 쓰지 않는다. |
| 카탈로그 컨텍스트 | 가능하면 **`works` 전체**를 LLM 컨텍스트에 포함한다. |
| 컨텍스트 초과 시 | **`works`를 토큰 한도에 맞게 여러 청크로 분할**하여 동일 상점에 대해 **여러 번 호출**하고, 결과를 **합친다**. |

## 비목표

- 상점 파이프라인이 매칭 실패분을 위해 **새 `works` 행을 자동 생성**하는 것.
- 미매칭 작품을 위한 **운영 큐 UI** (후속 과제로 분리 가능).
- 본 문서만으로 **기존 `works` 행 병합·중복 정리**를 수행하는 것(수동 관리 전제).

## 권장 접근(타안 대비)

1. **통합 추론(권장)**  
   상점 메타·KB 요약·`work_id` 선택을 **한 스키마의 JSON 출력**으로 묶는다. 카탈로그가 크면 청크 호출·합성으로 처리한다. 맥락이 한 흐름에 남는다.

2. **2단계(추출 후 매칭)**  
   디버깅은 쉬우나 호출 수·비용이 늘고 단계 간 표현 불일치가 생길 수 있다.

3. **결정론 선매칭 + LLM 잔여**  
   비용에는 유리하나 “카탈로그 전체를 넣고 한 번에 추론” 전략과는 결이 다르다.

본 리빌드 시나리오에서는 **1)**을 기본으로 한다.

## 아키텍처·데이터 흐름

1. **`works` 스냅샷**  
   처리 시점의 `works` 테이블에서 매칭에 쓸 컬럼(예: PK `id`, `name`, `korean_title`, `title_romaji`, `title_english`, `title_native` 등 운영 중 실제로 채워진 필드)을 읽어 **LLM용 목록**을 만든다. 값이 NULL인 필드는 후보에서 제외한다.

2. **상점 처리**  
   크롤 텍스트 수집 → LLM 호출(통합 프롬프트) → JSON 파싱.

3. **검증**  
   응답에 포함된 모든 `work_id`(또는 PK)가 스냅샷에 존재하는지 검사한다. **스냅샷에 없는 ID는 버린다.** 검증 통과한 ID만 `shop_works`에 반영한다.

4. **청크**  
   스냅샷 전체가 한 프롬프트에 들어가지 않으면 `works` 행을 **연속 블록** 등으로 나누어 각 청크마다 동일한 상점 컨텍스트로 호출한다.

5. **합성**  
   각 청크가 반환한 `work_id` 집합의 **합집합**을 최종 후보로 한다. 동일한 원문 언급이 서로 다른 ID로 매핑된 경우(드묾)에는 **결정적 규칙 하나**로 단일화한다(예: 더 작은 `work_id` 채택, 또는 첫 성공 청크 우선). 충돌은 **로그**로 남긴다.

6. **리빌드 범위**  
   상점 쪽 테이블(`shops`, `shop_details`, `shop_categories`, `shop_works`, `shop_links` 등 실제 사용 스키마에 맞게)은 **비운 뒤 파이프라인으로 재삽입**한다. **`works`는 비우지 않는다.**

## LLM 출력·오류 처리

- 출력은 **순수 JSON**, 구조는 구현 시 `refine_service` 관례와 맞춘다.
- **가챠·피규어 비관련** 상점은 기존처럼 `is_figure_relevant: false` 등으로 두고 RDB 저장 생략 가능.
- API·네트워크 오류는 기존 파이프라인과 동일하게 **실패·재시도** 정책을 따른다.

## 테스트·검증(구현 시)

- 스냅샷 빌더: NULL 필드 제외·빈 문자열 처리.
- 검증기: 존재하지 않는 ID 제거.
- 청크 합성: 두 청크 결과 합집합 및 충돌 규칙 단위 테스트.
- 짧은 크롤 픽스처로 한두 상점 **스모크**.
